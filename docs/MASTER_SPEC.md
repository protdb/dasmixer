# DASMixer: Приложение для совмещения идентификаций пептидных последовательностей в данных масс-спектрометрии

Ключевой функционал: слияние данных идентификаций для de novo assisted поиска, проведение полного цикла сравнительного анализа протеомных идентификаций

# Перечень возможностей

- Загрзука данных:
  - Загрузка данных спектра (.MGF, .MZML (?))
  - Загрузка данных идентификации пептидов и белков с использованием библиотечных методов (PLGS, PeptideShaker, MaxQuant(?))
  - Загрузка данных идентификации de novo (PowerNovo 1, 2, pi-Prime Novo, CasaNovo(?), другие(?))
  - Загрузка многих файлов идентификации и многих файлов MGF в рамках одного проекта
  - Назначение группы сравнения файлам для проведения сравнительного анализа, анализ многих групп (>2)
- Обработка идентификации на уровне пептидов:
  - Объединение данных, установка treshold'ов, правил выбора идентификации спектра, критериев корректной идентификации
  - Соотнесение с библиотечными данным для de novo, включая возможность импорта идентификаций (PowerNovo2). Расчет PPM и b/y для наложения моделей
  - Выбор оптимального варианта идентификации из результатов работы нескольких решений, по % интенсивности покрытия спектра ионами заданных типов (в т.ч. оптимальная пара: a/x, b/y, c/z (?))
  - Графики совпаденя ионов (abcxyz), настройка treshold'ов ионов, сравнительные результаты распознавания несколькими системами
  - оценка совпадения ppm
- Функционал работы на уровне белков:
  - Сопоставление с библиотекой белковых последовательностей, настраиваемое по критериям, в т.ч. % неполного соответствия для De Novo. Возможность использования белковых идентификаций библиотечного метода из загруженного файла библиотеки. Валидация совпадения по площади при неполном соответствии
  - Обогащение данных белковой идентификации по UniProt
  - LFQ-функционал: emPAI, iBAQ, TPA, Top3, NSAF, другое(?) - также как отделньая библиотека semPAI
- Результаты и выгрузки:
  - FC/p-val
  - Volcano plot
  - PCA
  - Достоверные PTM
  - Достоверные аминокислотные замены (denovo)
  - Выгрузки XLSX, mztab, объединенные и пофайловые
  - Качественные идентификации denovo без соотнесения белку в библиотеке
  - Сохранение графиков с выбором формата (png/svg/plotly JSON)

# Технические требования

- Кросс-платформенное десктопное приложение, работает на Windows, Linux, (потенциально - Mac OS, сборка будет доступна если удастся найти макбук)
- Распространение: .MSI-сборка, установка как пакет python через PyPi, conda(?), snap.
- Использование многопоточности при обработке
- Хранение пользоватлеьских настроек: пути чтения/сохранения файлов должны переиспользоваться при открытии вормы загрузки/сохранения, обращение должно всегда идти по последнему использованному под задачу пути
- использование системного пути (appdata и т.п.) для хранения конфигурации
- Весь интерфейс представлен на английском языке, но должна быть заложена возможность использования локализации сравнительно простым способом.

# Понятие проекта

Проект - внутренняя сущность, содержащая все данные исследования. Хранится одним файлом, являющимся БД SQLite. Струкутра данных в рамках проекта представлена на диаграмме PROJECT_ER.mermaid

В проекте хранится вся основная информация, начиная от содержимого MGF-файлов, заканчивая данными созданных отчетов. Ключевые моменты работы с проектом:
- Большинство параметров nullable и могут быть не заполнены.
- Сохранение спектров в файле: сохраняется массив numpy (одномерный) как python pickle сжатый в gzip
- Если для спектра используется одно значение заряда для всех mz, то задается значение charge_array_common_value, равное этой величине, а charge_array становится null

С точки зрения API Project - это объект, который может быть создан с нуля, открыт (передается путь), сохранен (задается путь), "сохранен как" (задается альтернативный путь). Может работать контекстным менеджером (with...) при обязательной передаче пути при открытии (в таком случае в конце выполняется сохранение проекта).

В сущности Проект инкапсулируется вся работа с БД, включая все основные представления данных. должен быть предусмотрен низкоуровневый API к запросам БД, но как вспомогательная функция для сложных отчетов. при этом Проект отвечает только за хранение данных и доступ к ним. Основной формат данных, предоставляемый и принимаемый Проектом - Pandas DataFrame. Построчные данные возвращаются уже в других форматах, в зависимости от специфики. Для сложных сущностей, используемых часто и обрабатываемых поодиночке (Tool, Subset) нужны dataclasses, используемые как внешний интерфейс вместо взаимодействия с БД.

Подразумевается основной сценарий работы:
1. Проект отдает данные
2. Внешняя (по отношению к Проекту) логика их обрабатывает
3. Проект сохраняет данные

Проект также является сущностью, работающей асинхронно, т.е. все его методы являются асинхронными во избежание блокировок UI. В перспективе имеет смысл сделать синхронный Wrapper над Проектом, но не на первом этапе разработки.

# Типовые компоненты

Ряд элементов решения является типовыми, т.е. могут быть реализованы через универсальный интерфейс. В частности это:
- Парсер данных спектров (MGF, MZML)
- Парсер данных идентификации
  - Парсер идентификации в табличном виде, .csv, .xls, .xlsx
    - Парсеры для разных вариантов выгрузки (PLGS, PeptideShaker, SEQUEST)
  - Парсер идентификаций других форматов (MzTab? В будущем)
- Метод полуколичественного анализа (детали ниже)
- Модули отчета. Каждый включает: логику получения данных из Проекта, форму Flet для настройки параметров (и список параметров для CLI), вывод (один или более из): текст, диаграмма (Plotly go.Figure), датафрейм

Для этих элементов создаются абстракные классы, которые далее имплементируются под конкретный случай.

# CLI

Логика работы CLI следующая:
- Если запускается без параметров: открывается окно приложения
- Если запускается с параметром пути к файлу: открывает переданный файл как проект
- Можно указать путь и команду. Для команды задаются параметры (в зависимости от команды):
  - add/drop, флаги: --tool, --subset (добавить/удалить инструменты и группы в проекте)
  - mgf: импорт MGF, параметры аналогичны настройкам
  - import: импорт файлов в заданный инструмент
  - report: подготовка отчета


# Архитектура решения

- UI на Flet
- CLI на Typer
- Отрисовка графиков: plotly с интерактивным режимом в pywebview
- Работа с данными: pandas, numpy
- Протеомные данные: pyteomics, peptacular, npysearch
- Управление проектом: poetry
- Работа с собственным форматом файла: SQLite
- Настройки: pydantic-settings
- Выгрузка данных, поддержка форматов: kaleido, openpyxl, xlsxwriter...

Внутренняя архитектура разделяет логику и интерфейс, при этом представляется большинство функций в виде трёх интерфейсов:
- GUI
- CLI
- Python API (т.е. использование приложения как пакета Python)

При этом уровень python API используется внутри приложения, но должен быть доступен извне при установке в качестве пакета python


Струкутра исходного кода формируется следующим образом:


- api
  - project # работа с данными пользовательских проектов
    - models # модели данных для работы с SQLite-структурой проекта
    - dataclasses # датаклассы для строгого форматирования данных
  - inputs
    - spectra # входные данные, MGF, MzML
    - peptides # табличные данные для работы с пептидами
    - proteins # данные белков
      - fasta # загрузка FASTA
      - enrichment # загрузка обогащающих данных (UniProt)
  - spectra
    - ions # оберткая над функционалом работы с ионами: генерация фрагментов, разметка b/y ионов, расчет процента покрытия для последовательности
    - validation
      - ppm
      - sequence # преобразование нотаций, работа с ProForma
    - selection
  - protein
    - matching # поиск белков по совпадению последовательности
    - lfq # label-free квантификация, детализация TBD
    - statistics # расчет статистических значений в рамках сравнительного анализа
  - reporting # отчеты, графики и выгрузки
    - pca
    - volcano
    - upset
    - diffexp # дифференциальная экспрессия белков и групповая специфичность
    - proteoforms # отчет о выявлении протеоформ: de novo идентифицированные аминокислотные замены, идентифицированные ПТМ
- gui # представления пользовательского интерфейса
  - app.py # исходная точка GUI, роутинг, меню, панели.
  - empty # страница создания проекта / выбора проекта
  - project # представление проекта
    - container # контейнер вкладок
    - tabs # содержит вкладки основного UI проекта
      - samples
- cli # представления интерфейса командной строки

# Особенности проекта и использование Агентов

- Документация для пользователя ведется на английском языке. Документация разработки (описание изменений, спецификации) ведется на русском; взаимодействие между Агентом и разработчиком ведется на русском;
- Тестирование: не генерируем тестовые данные, они будут предоставлены разработчиком; не используем юнит-тестирование, используем интеграционное; тесты пишем всегда отдельным шагом, по команде, не вместе с кодом.
- Информация о разработке проекта: Laboratory of Structural Proteomics, IBMC, Moscow. GitHub: git@github.com:protdb/dasmixer.git, Название проекта: DASMixer
- При необходимости зависимости добавляются без указания версии (текущая наиболее актуальная)
- Распределение ролей: значительная часть компонента API пишется Разработчиком на основе имеющихся наработок в рамках разрозненных проектов. В целом, одна из вспомогательных задач этого проекта - собрать многие наработки по сравнительному анализу протеомных данных и интеграции данных de novo секвенирования с библиотечными данными в продукт, доступный широкому кругу пользователей. Агент не изменяет pyproject.toml без острой необходимости
- Стандартная схема действий, Asc-Plan-Act-Document. При работе Агент сначала задает уточняющие вопросы, далее формирует спецификацию (поля UI, спецификации функций, разделение по файлам и т.д.), далее этот план согласовывается с разработчиком и запускается в работу. После подтверждения от Разработчика формируется пользовательская и техническая документация
- Документация строится по следующей схеме:
  - Пользовательская
    - CLI
    - API
      - Включая описание доступных функций, классов и методов
    - GUI
  - Техническая:
    - Описание внутренней архитектуры
    - Описание компонентов и логики их взаимодействия
    - Описание деталей реализации
  - Проектная
    - Данный документ
    - Спецификации на разработку
    - Описание изменений в рамках конкретной сессии разработки

## План разработки:
1. Первый этап (выполняется Агентом):
   1. Структура проекта
   2. Класс Project, его ключевые внешние интерфейсы для добавления данных
   3. Абстрактные классы для импорта спектров и результатов идентификации
   4. Абстрактный класс для отчетов
2. Второй этап (Выполняется преимущественно разработчиком):
   1. Конкретные импортеры для MGF, PowerNovo 2, MaxQuant
   2. Функционал работы с идентификациями в API: разметка ионов, расчет метрик качества идентификаций
   3. Функции построения графиков разметки ионов
   4. Функционал поиска файлов и определения ID образца
   5. Выгрузка идентификаций спектров
3. Третий этап (Агент):
   1. Базовый GUI (окно нового проекта, страница импорта данных)
   2. GUI для просмотра сопоставлений пептидов (вкладка peptides, кроме отрисовки графиков)
   3. Базовый CLI (создание/открытие проекта, добавление спектров и идентификаций)
   4. Интеграция процесса, применение функций API в GUI и CLI
4. Четвертый этап (разработчик)
   1. Поиск белковых идентификаций (API)
   2. Обогащение данных (UniProt, ProteinAtlas)
   3. Основные отчеты
   4. Интеграция pywebview для интерактивного просмотра графиков
   5. Функционал полной выгрузки отчетов по списку, формирование шаблона документа
5. Пятый этап (Агент)
   1. Интеграция 4-го этапа, GUI для белковых идентификаций и для отчетов
   2. Отображение графиков из интерфейса через pywebview
   3. Финализация, нехватающие доработки

В конце каждого этапа выполняем тестирование функционала, также исправляем по необходимости баги. Для функционала API и CLI пишем тесты. К проекту будут приложены тестовые данные.
