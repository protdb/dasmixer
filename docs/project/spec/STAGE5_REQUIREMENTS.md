# Этап 5: отображение и рендер отчетов

Отчеты - это модули, которые выполняют детальный анализ данных.
На вход отчету подаются:
- project
- параметры, специфичные для отчета

Основной метод для отчета: generate_report(), который принимает на вход параметры и возвращает фиксированную структуру данных вида:
```python
tuple[list[tuple[str, go.Figure]], list[tuple[str, pd.Dataframe, bool]]]
```
Которая состоит из:
- список графиков в виде кортежей из:
  - названия графика
  - представления графика как объекта plotly.graph_objects_figure
- список табличных результатов в виде кортежей из:
  - названия таблицы
  - pandas DataFrame с данными таблицы
  - признака отображения таблицы в UI/документах (если False, то таблица сохраняется только при экспорте в Excel)

С отчетом возможны следующие действия:
- Отображение в интерактивном режиме: из данных формируется HTML-страница, включающая таблицы (с признаком короткой таблицы) и графики (как plotly.json внутри `<script>`)
- Выгрузка в текстовом формате: формируются документы в форматах docx (контент аналогичен HTML-документу), .xlsx (содержит страницы отчета на листах)
- Массовая выгрузка нескольких отчетов в один документ/одну таблицу

В любом варианте выгрузки включается информация о:
- Параметрах проекта
- Параметрах всех инструментов
- Группах (название, описание, кол-во образцов)

Для настроек параметров в пользовательском интерфейсе сейчас делаем одно текстовое поле, содержащее все необходимые параметры со значениями по умолчанию в формате:
```
PARAM1=VALUE1
PARAM2=VALUE2
```
Т.е. с новой строки с разделением через '='

Отчеты должны предоставлять словарь с параметрами и их значениями по умолчанию, т.е. у отчета должен быть
staticmethod get_parameter_defaults, возвращающий `dict[str,tuple[Type,str]]`, где будут название параметра,
тип, к которому должно быть приведено значение, дефолтное значение в виде строки. Пример:
```python
{
    'tool1': (str, 'PN2'),
    'tool2': (str, 'MQ'),
    'max_values': (int, '42'),
    'some_check': (str, 'Y') #Не делаем сейчас обработку bool, тут сразу сложности начнутся, а решение с текстом временное
}
```

Такой UI позволяет упростить задачу. В идеале хорошо бы использовать генерацию формы flet на основе dataclass'а
или модели pydantic, но сейчас это слишком сложно.

Для выгрузки, любой, должен быть предусмотрен метод get_context(), который является методом базового класса
BaseReport, не требующим override'а. Возвращает dict со следующей структурой:
```python
context = {
    "settings": {
        "project_parameters": [
            {
                "key": "str",
                "value": "str"
            }
        ],
        "tools": [
            {
                "name": "tool name",
                "type": "library/denovo",
                "settings": {...} # tools.settings из project
            }
        ],
    },
    "show_parameters": True,
    "report_name": "str",
    "report_parameters": [
        {"key": "str", "value": "str"}
    ]
    "tables": [
        {
            "name": "str",
            "data": {
                "headers": ["str", "str", "str"],
                "rows": [
                    ["val1", "val2", "val3"]
                ]
            }
        }
    ],
    "figures": [
        {
            "name": "str",
            "png": b'bytes_of_png_image',
            "json": "plotly_json_as_str"
        }
    ]
}
```

HTML-страницу отрисовываем с помощью jinja2, для шаблона word пока сделаем пустой метод, я его напишу сам, но в нём используется тот же контекст

Результаты также сохраняются в БД, в таблицу `generated_reports`. Сохраняются и выгруженные данные, и графики, 
и параметры системы в момент расчета. Для данных используем pickle и gzip. Таблицу generated_reports переделываем совсем,
а именно, в ней должны быть поля:
- id (INT PK AI)
- report_name (название отчета)
- created_at (дата создания)
- plots (BLOB с графиками)
- tables (BLOB с табилцами)
- project_settings (параметры проекта на момент создания, JSON в строке)
- tools_settings (параметры инструментов)
- report_settings (параметры отчета на момент запуска)

Также добавляется таблица report_parameters с параметрами отчетов, заполненными пользователем. Сохраняется при каждом запуске, содержит поля:
- id
- report_name
- parameters (TEXT с параметрами в формате `key=value\nkey2=value` как в форме в GUI)

Требуется разработать registry для отчетов, аналогично таковому для inputs. Отчеты в перспективе тоже могут быть вынесены в плагины

Для отчета должна быть опция инициировать его уже с данными, без вызова generate.

generate в базовом классе должна быть в обертке, которая обрабатывает данные на входе и на выходе, сохраняет внутри себя и в БД.

GUI для отчетов:
состоит из двух блоков:
- Общие настройки:
  - Размер шрифта графиков
  - Высота изображений
  - Ширина изображений
  - Кнопка "Generate all" (генерирует все отчеты)
  - Кнопка "Export all"
- Полный список отчетов, для каждого:
  - Название
  - Чекбокс "generate with all" (по умолчанию вкл)
  - текстовое поле с параметрами
  - кнопка "Generate" (сгенерировать и сохранить)
  - выпадающий список с датами и временем сохраненных в проекте генераций отчета
  - кнопка "View" (открывает интерактивный режим просмотра)
  - кнопка "Export" (открывает выбор папки для сохранения)

Для формирования отчета в Word используем модуль `docxtpl`, сейчас для экспорта должен быть метод-заглушка 
в базовом классе отчета, который получает на вход путь до папки для сохранения. Там же будет экспорт в excel.
