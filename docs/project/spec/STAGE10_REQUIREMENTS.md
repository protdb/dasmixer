Этап 10, доработки и модификация порцесса работы с белковыми идентификациями; интеграция улучшенного процесса работы с ppm для белков

Доработки затрагивают логику во вкладке peptides в UI (`gui/views/tabs/peptides/peptides_tab_new.py`), связанную с мэппингом белков из БД идентификаций на пептиды.

Ориентируемся как на основу на то, как реализована логика для идентификаций, см. `api/calculations/spectra/identification_processor.py`, переиспользуем по максимуму то, что касается маппинга (внешние вызовы из функций), но в целом делаем всё аналогично

Изменения в логике следующие:

- сейчас реализованы два отдельных процесса:
  - Match peptides to proteins
  - Calculate PPM and coverage for proteins
- В раках первого выполняется маппинг пептидов к белкам (см. `api/calculations/peptides/protein_map.py`), а в рамках второго - для отличающихся идентификаций выполняется рассчет ppm и мэтчинга ионов
- Необходимо объединить эти шаги:
  - В рамках мэппинга пептидов: если Identity после Blast равна 1.0 (последовательности совпадают), то нет необходимости считать ppm и ion coverage, нужно просто скопировать из identification
  - Если есть отличие в последовательности, то необходимо пересчитать для найденной последовательности PPM и параметры мэппинга фрагментов. При этом необходимо использовать SeqFixer (`api/calculations/ppm/seqfixer.py`), метод `get_matched_ppm` в который переданы параметры, рассчитанные для идентификации, и последовательность в исходном виде.
  - Добавляется новый параметр, настраиваемый в tool_settings (блок из нескольких checkbox'ов): Match correction criteria. Включает варианты:
    - ppm
    - intensity_coverage
    - ions_matched
    - top10_ions_matched
  - При identity < 1.0 мэтч пептида к белку считается корректным, если хотя бы один из выбранных пользователем критериев не ниже (больше или равно для всех критериев, кроме ppm; для ppm, соответственно, меньше или равно) значению для исходной последовательности
  - В project в таблицу peptide_match добавляются параметры matched_peaks, matched_top_peaks, matched_ion_type (те же данные, что в ions_matched, top_peaks_covered, ion_matchtype в identification соответственно). И туда же добавляем matched_sequence_modified, которая заполняется только если get_matched_ppm вернул варианты с модификациями. НЕ делаем миграции/fallback, сейчас это совершенно не нужно.

С точки зрения UI вносим следующие изменения:
- Убираем флаг match preferred only в блоке Protein mapping settings, он не нужен
- Убираем отдельную кнопку Calculate PPM and coverage for proteins из блока actions, также не делаем соответствующий вызов при нажатии Calculate petides

В рамках процесса в protein_map.py у нас остается логика:
- Сбор последовательностей в batch, поиск по БД с учетом входных параметров
- Объединение данных идентификаций
- При полном совпадении canonical_sequence и matched_sequence - не выполняем расчёт ppm, используем значения от identification
- При неполном совпадении - используем новую методологию
- Добавляем неполные совпадения только если результат по выбранным критериям не хуже, чем у исходного пептида

В этой логике мы добавляем ещё одну петлю дополнитлеьного сценария:
- Добавляем для tool флаг "Save AA substitutions"
- В таблицу peptide_match добавляем ещё параметр: substitution
  - Если флаг установлен, то:
    - Сохраняем с этим флагом даже матчи с неполным совпадением (identity < 1.0) и худшими, чем у identification, Match correction criteria, при условии, что identification соответствует ограничениям по длине, покрытию и ppm, установленным для инструмента.
