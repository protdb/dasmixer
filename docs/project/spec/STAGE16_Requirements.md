# Этап 16. Доработки импорта данных

## Импорт ID белков при импорте идентификаций

Большинство библиотечных решений сразу же в файле с идентификациями передают идентификаторы белков. Надо этим воспользоваться.

Добавляем в таблицу identifications поле src_file_protein_id (TEXT, nullable).

В IdentificationParser (`dasmixer/api/inputs/peptides/base.py`) добавляем следующую логику:
- отдельный атрибут _proteins: `dict[str,Protein]`, в котором хранятся данные белков
- отдельный атрибут `contain_proteins: bool = True`
- в init два флага: `collect_proteins, is_uniprot_proteins`

Логика (должна быть описана в руководстве по плагинам и в docstring'ах):
- По мере обработки файла идентификаций, информация о белках собирается в _proteins в формате protein_id: ProteinObject
- При сохранении идентификаций в parse_batch идентификациям добавляется src_file_protein_id c ID белка
- После завершения чтения файла, в БД сохраняются белки c `ON CONFLICT(id) DO NOTHING;` (т.е. не перезаписываем существующие).
- Важный нюанс - если передан is_uniprot_proteins, то всем объектам Protein проставляем is_uniprot=1 иначе 0.
- Флаг collect_proteins влияет только на то, добавляются ли белки в БД.

Также в силу описанной выше логики и работы с protein, убираем из спецификации и имплементаций метода parse_batch второй pd.DataFrame, который по старой задумке должен был содержать данные о белках

Доработки в `dasmixer/api/inputs/peptides/table_importer.py`:
- src_file_protein_id добавляем в ColumnRenames
- Отдельный метод `get_proteins(self, row: df: pd.DataFrame) -> dict[str:Protein] | None`, который вызывается для каждого батча. Если результат is not None, то обновляем self._proteins

Изменения UI:
- Добавляем упомянуты флаги в диалоге импорта идентификаций (активны при условии, что для класса парсера `contain_proteins=True`)

Изменения в логике поиска идентификаций:
- Если стоит флаг Use protein ID from file: перед поиском с npysearch для всех идентификаций, у которых src_file_protein_id is not None читаем это поле; получаем из него идентификатор/список идентификаторов (может быть несколько через ";") и для каждого из этих идентификаторов создаём запись в peptide_match с точным совпадением последоватлеьности.
- После этого выполняем поиск только для идентификаций, у которых этого значения нет.

## Импорт идентификаций для нескольких спектров из одного файла

Допускается и вероятна ситуация, когда для несклоьких sample_id и/или spectre file идентификации лежат в одном общем файле. Для этого случая необходимо предусмотреть альтернативный подход к получению данных

Схема следующая: в диалог import identifications добавляется ещё одна кнопка (только туда, для спектров такой подход не поддерживаем!) Import stacked file

Открывается дополнитлеьный диалог, в котором 1) выбираем файл, нажимаем кнопку get samples list.

Также добавляем поле Sample field (позволяет сделать override поля с образцами)

Отображаем список значений столбца с образцами, аналогично таковому при импорте по паттерну. Значения должны совпадать с названиями существующих в БД образцов.

Изменения в БД: для identification_file добавляем два необязательных поля:
- selection_field
- selection_field_value

В эти поля добавляем значения, выбранные в диалоге.

Т.е. технически на каждый образец, представленный в общем файле, создаются отдельные записи в identification_file с одинаковым путём. Но мы сохраняем критерий, по которому был отобран данный образец

На стороне IdentificationParser:
- параметр can_import_stacked: bool = False
- параметр sample_id_column: str | None = Mone для отображения
- метод get_sample_ids(self, override_column=None) -> list[str] для получения списка образцов (путь передается в init)

parse_batch должен возвращать sample_id среди прочих столбцов, но если can_import_stacked = False, то игнорируем даже его наличие.

Дополняем этой возможностью парсер MaxQuant, там поле называется "Raw file"
