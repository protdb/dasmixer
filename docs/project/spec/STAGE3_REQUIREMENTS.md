Делим задачи на два подэтапа:

Этап 3.1. Доработки первой и второй фазы.

Сейчас не паримся обратной совместимостью, вносим важные правки, выявленные в процессе и документируем их:
- в таблице spectre вносим также параметр intensity
- для парсера спектров давай уберем обязательную metadata, включая определение количества спектров. Для определения общего количества файл обычно надо целиком прочитать, что долго и бессмесленно. Нет смысла выделять их до парсинга
- файлы идентификации пептидов для отдельных инструментов могут также содержать идентификации белков. Это важный момент, т.е. в одном файле может быть два разных результата. В этом случае предусмотрено, что parse_batch может вернуть дополнительно ещё один массив с идентификациями белков. Пока просто учитываем эту логику, Реализуем корректную работу с ней позднее.
- Парсерам совершенно незачем знать, какие у них ID файла и инструмента, этим нужно обогащать идентификации уже после получения из парсера, в рамках логики работы процесса в целом (UI/CLI)
- при загрузке идентификации должно быть необязательно знать, какой id у спектра в БД. на уровне логики проекта должен быть маппинг либо по scans, либо по spectra_no. Сейчас в парсерах реализована логка преобразования (в некоторых), её необходимо изменить и сделать в проекте метод get_spectra_idlist, выполняющий корректный мэппинг либо по scans, либо по seq_no для конкретного файла идентификаций.
- Я выполнил бОльшую часть задач этапа 2, сейчас в коде реализован ряд модулей, обеспечивающих парсинг данных, а также простые расчеты и обработку. Твоя задача сейчас - задокументировать их и поверхностно проверить корректность кода, вывести рекомендации по улучшению
- Нужно реализовать логику регистрации парсеров в inputs.registry
  - Регистрация должна выполняться в отдельном вызове при запуске, должны регистрироваться полноценные системные идентификации модулей, также сейчас стоит предусмотреть (но не реализовывать!) создание в дальнейшем функционала плагинов для добавления парсеров
