## Миграции Project

В метаданных проекта (таблица project_metadata) есть параметр version. При открытии проекта нужно проверять, что значение version равно актуальной версии файла проекта. Если значение ниже, то используем логику миграции

создаем модуль dasmixer.api.project.migrations, в нём должны быть:

Константы:
```python
MIN_SUPPORTED_VERSION = "0.1.0" # Минимальная версия, для которой возможна миграция. Проект ниже указанной версии даже не пытаемся обновить
CURRENT_VERSION = "0.2.0" # Текущая версия ПРОЕКТА (не приложения), до которой обновляем и которую сохраняем в Project Metadata
```

Функции:
```python
def apply_migrations(project: Project) # Опционально: сделать логику миграций как Mixin проекта
```

Принцип работы с версиями:
- версия файла проекта равна последней версии приложения, в которой были внесены изменения в схему проекта.
  - То есть: сейчас мы вносим изменения в приложение, поднимаем версию до 0.2.0, изменения затрагивают схему проекта -> поднимаем версию проекта до 0.2.0
  - Потом в версии 0.3.0 мы внесем изменения, не затрагивающие схему проекта. Актуальная версия проекта останется 0.2.0
  - Потом в версии приложения 0.4.0 мы внесем изменения в схему проекта - и поднимем версию проекта до 0.4.0

Логика применения миграций:

Миграции хранятся в следующем формате:
```python
MIGRATIONS = [
    {
        "version": "0.2.0",
        "migration": "SQL TO MIGRATE PROJECT FILE GOES HERE..."
    }
]
```

Соответственно, apply_migrations просто идет по списку, пропустив в начале миграции для версий ниже или равные версии открытого проекта и применяет миграции к следующей версии до конца (последняя: та, которая равна текущей версии проекта)

Логика пользовательского интерфейса:

Версия проверяется сразу после открытия проекта. При этом мы не блокируем работу с проектом неподходящей версии:
- Если версия проекта ниже текущей версии:
  - Уведомляем пользователя в модальном окне. Описываем ситуацию: если пользователь не обновит проект, он может нарваться на ошибки в текущей версии DASMixer. Если обновит - могут возникнуть проблемы в старых версиях
  - Пользователь может согласиться обновить или отказаться. При согласии: модальное окно с прогрессом, потом - snack_bar "Project updated to {version}"
  - При отказе - не спрашиваем до следующего открытия проекта
- Если версия выше: просто показываем модальное окно с предупреждением о возможных ошибках в случае использования проекта более высокой версии.
